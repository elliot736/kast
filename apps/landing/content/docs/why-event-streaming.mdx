Most job monitors work like a camera that takes a photo every few minutes. You get snapshots: "monitor X was healthy at 3pm. Down at 3:05pm. Healthy again at 3:20pm." If you want to understand what happened between those snapshots, you are guessing from timestamps.

Kast works like a video camera. It records every event as it happens, in order, and keeps the full tape. This is not an implementation detail you can ignore — it directly changes what you can do when something goes wrong.

## Everything Is an Event

Every action in Kast produces a durable, ordered event:

- A ping arrives → `PingReceived` event
- The sweep detects a missed ping → `MonitorStateChanged` event
- An incident is opened → `IncidentOpened` event
- An alert is dispatched → `AlertDispatched` event
- Delivery to Slack fails → `AlertDeliveryFailed` event
- The retry succeeds → `AlertDelivered` event
- Your job recovers → `IncidentResolved` event

These events are stored in an ordered log (powered by Redpanda, a Kafka-compatible streaming platform). Each event has a timestamp, a sequence number, and all the data about what happened.

Compare this to a typical monitoring tool that stores rows in a database: "monitor X, status healthy, updated_at 3:00am." When the status changes to "down," the previous row is overwritten or a new row is inserted. You know the current state, but the intermediate events — the exact moment the grace period expired, the exact moment the alert was dispatched, whether delivery failed and was retried — are lost or scattered across different tables.

In Kast, nothing is overwritten. Events are appended to an immutable log. The database is a derived view built from that log. If the database disappeared tomorrow, you could rebuild it by replaying events from the beginning.

This gives you four capabilities that snapshot-based monitors can't provide.

## Backtest Alert Rules

You changed the grace period from 5 minutes to 10 minutes. Would last week's 3am alert still have fired?

With a snapshot-based monitor, you would have to wait for the next incident to find out. With Kast, you replay the events through the new rules:

> The ping was expected at 03:00. With a 10-minute grace period, the deadline would have been 03:10. The job recovered at 03:35, so yes — the alert would still have fired, but 5 minutes later.

Now try the opposite: you're getting too many alerts and want to increase the grace period to 30 minutes. Replay last month's events:

> Out of 4 incidents, 3 would have auto-resolved within 30 minutes (the job was just slow). Only 1 would have triggered an alert — the actual outage. False alerts reduced by 75%.

You can make this decision with data instead of guessing.

<Callout type="info">
Backtesting works because events are immutable and ordered. Kast replays the exact sequence that occurred, substituting your new rules for the ones that were active at the time. The result tells you what would have happened.
</Callout>

## Ordered and Consistent

One property of the event log that matters in practice: events are ordered. When you look at an incident timeline, the events appear in the exact order they occurred. There is no ambiguity about whether the alert was dispatched before or after the acknowledgment.

This ordering is guaranteed by Redpanda's partition-level ordering. All events for a single monitor go to the same partition, so the sequence within a monitor is always correct. You will never see a timeline where the resolution appears before the detection.

<Callout type="info">
Ordering matters most during postmortems. When someone asks "did the alert fire before or after the deploy?", the event log gives you a definitive answer — not an approximation based on timestamps that might have clock skew.
</Callout>

## Zero-Poll Dashboard

In most monitoring tools, the dashboard polls the API every 15-30 seconds. You are always looking at stale data. "Is this current? Let me refresh." You click refresh. Nothing changes. You click again. Still nothing. Then suddenly it updates — 20 seconds after the actual event.

Kast's dashboard doesn't poll. Events flow directly from the event log to your browser over WebSockets:

```
Redpanda → WebSocket server → your browser
```

When a ping arrives, you see it instantly. When a monitor goes down, the badge turns red within a second. When an incident resolves, it updates live. No refresh button. No "last updated 30 seconds ago" label. No wondering if the data is stale.

This matters at 3am when you are staring at the dashboard trying to figure out if the fix you just deployed actually worked. You don't want to wait 30 seconds for the next poll cycle. You want to see the success ping appear the instant it arrives.

## Instant Recovery

When Kast restarts — whether from a deploy, a crash, or a server reboot — it needs to know the current state of every monitor. Is `db-backup` healthy or down? Is there an open incident? When is the next expected ping?

Snapshot-based systems query the database and hope the data is current. Kast reads the compacted `monitor-state` topic from Redpanda. This topic contains the latest state for every monitor, and Kast can read the entire thing in seconds.

The result: Kast goes from cold start to fully operational in under 5 seconds, regardless of how many monitors you have. No stale state. No missed alerts during restart. No "the monitoring was down for 2 minutes during the deploy and we missed an incident."

## What This Means for You

<ImageZoom src="/diagrams/dataflow.png" width={870} height={1048} alt="Data flow diagram showing events traveling from ping ingestion through Redpanda topics to schedule evaluation, incident management, alert dispatch, and real-time streaming" />

Here's the practical impact:

**You never lose information.** Every event is retained in the stream for 7 days, then lives in PostgreSQL for as long as you need. There is no gap between "the event happened" and "we recorded it." The event log is the source of truth.

**You can reconstruct any incident after the fact.** No more "I think the backup failed around 3am but I'm not sure." Open the replay, see the exact sequence of events, and know precisely what happened.

**You can test changes safely.** Before changing a grace period, a cooldown, or an alert rule, backtest it against historical events. Know the impact before you deploy.

**Your dashboard is always live.** Events flow from the log to your screen in real-time. No refresh button. No polling delay. No stale data.

**Restarts are invisible.** Kast recovers its full state from the event log in seconds. You never miss an alert because the monitoring system was restarting.

**Adding new integrations is safe.** Because each consumer reads from the log independently, adding a new alert channel or metrics exporter doesn't affect existing functionality. New consumers start reading from the log without changing any existing code.

**Auditing is built in.** Every action is recorded. If someone asks "why did we get paged at 3am?", you can trace the exact event chain: this ping was expected, this grace period expired, this incident was opened, this alert rule matched, this channel was dispatched to. There is no guessing.

## Event Retention

Events live in two places:

- **Redpanda stream** — 7 days of retention. This is the hot path: real-time consumers, WebSocket streaming, replay, and backtesting all read from here. The stream is fast and ordered.
- **PostgreSQL** — Permanent storage. The sink consumer writes every event to the database in batches. This is the cold path: dashboard queries, historical reports, and long-term auditing read from here.

The 7-day stream retention means you can replay and backtest any incident from the past week directly from the stream. For older incidents, the data lives in PostgreSQL and is available through the dashboard — you just can't stream through it at the same speed.

For most teams, this is the right tradeoff. The incidents you need to replay urgently are the ones that just happened. By the time something is a month old, you're looking at it in a postmortem document, not streaming through it in real-time.

<Callout type="info">
You don't need to understand Redpanda or Kafka to use Kast. The event-driven architecture works behind the scenes. What you see is a monitoring system that is fast, accurate, and never loses data.
</Callout>
