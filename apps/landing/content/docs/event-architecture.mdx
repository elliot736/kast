Kast is built on an event-driven architecture powered by Redpanda (Kafka-compatible). Every ping, state transition, incident, and alert is a durable event in a distributed log. PostgreSQL is a derived projection -- if you lose it, replay the log and reconstruct state.

## Topic Architecture

<ImageZoom src="/diagrams/topics.png" width={1693} height={521} alt="All 14 Redpanda topics with their producers and consumers" />

Kast uses 14 Redpanda topics across three pipelines: monitoring, job execution, and workflows. Each has a specific role, retention policy, partition count, and set of producers and consumers.

## End-to-End Data Flow

<ImageZoom src="/diagrams/dataflow.png" width={870} height={1048} alt="Ping event data flow from HTTP ingestion through schedule evaluation, incident management, and alert dispatch" />

A ping enters via HTTP, is published to `ping-events`, and cascades through the topic graph. Each stage reads from one topic and writes to the next, forming a directed acyclic graph of event processors.

## Event Sourcing Philosophy

<Callout type="info">
Redpanda is the **source of truth**. PostgreSQL is a derived projection optimized for queries. Every state change is recorded as an immutable event. The database can be rebuilt by replaying the event log. Adding a new integration (metrics exporter, webhook forwarder) means adding a new consumer group -- no changes to existing services. You can seek to any timestamp and replay events to reconstruct what happened at that point in time. Auditability is built in -- you can trace exactly why an incident was opened by reading the event log.
</Callout>

## Log Compaction for monitor-state

The `monitor-state` topic uses Kafka log compaction (`cleanup.policy: compact`). Redpanda retains only the latest message per key (monitor UUID), discarding older versions during compaction cycles.

This provides:

- **Instant recovery on restart.** Read the entire compacted topic on startup to hydrate the in-memory state for all monitors. No need to replay `ping-events` from the beginning.
- **Constant-size topic.** The topic size is proportional to the number of monitors, not the number of events over time.
- **Fast consumer initialization.** New consumers (including the WebSocket gateway on connection) can read current state in one pass.

## Partitioning by Monitor UUID

All topics that carry monitor-specific data are keyed by monitor UUID. This guarantees:

- **Strict ordering per monitor.** All events for a given monitor land in the same partition and are processed in order.
- **Parallel processing across monitors.** Different monitors can be processed concurrently by different consumer group members.
- **Predictable partition assignment.** Adding API replicas rebalances partitions across consumers without losing ordering guarantees.

## Tiered Data Lifecycle

| Tier | Storage | Retention | Access Pattern |
|------|---------|-----------|---------------|
| Hot | Redpanda topics | 1-30 days (per topic) | Real-time streaming, event processing |
| Warm | PostgreSQL | `PING_RETENTION_DAYS` (default 30) | API queries, dashboard, replay |
| Cold | Redpanda `monitor-state` (compacted) | Infinite | State recovery on restart |

Redpanda handles recent events for streaming and processing. PostgreSQL handles long-term queryable storage. The compacted `monitor-state` topic serves as a persistent snapshot that survives Redpanda retention cleanup on the other topics.

## Dead Letter Topics

When an alert delivery fails all 5 retry attempts, it is published to `alert-deadletter`. This topic is:

- Retained for 30 days.
- Exposed via `GET /api/v1/dead-letters` for inspection.
- Supports one-click retry via `POST /api/v1/dead-letters/:id/retry`.
- Surfaced in the dashboard dead letter page with delivery attempt history and error messages.

## Batch Processing in SinkModule

The Sink Module writes events to PostgreSQL in batches for performance:

- **Buffer size:** 50 events
- **Flush interval:** 200ms

Whichever threshold is hit first triggers a batch INSERT. This means a single ping event reaches PostgreSQL within 200ms in the worst case, or immediately if 50 events are buffered (high-throughput scenario).

### In-Memory Duration Tracking

The Sink Module maintains an in-memory `Map<string, Date>` that tracks `start` ping timestamps keyed by monitor ID. When a `success` ping arrives, the Sink looks up the corresponding `start` timestamp in the map, calculates the duration, and writes it to the ping record in PostgreSQL. This avoids a database round-trip to look up the start time.

The map is bounded by the number of active monitors, not the number of events, so memory usage is predictable. If the process restarts, in-flight durations for pings that received a `start` but not yet a `success` are lost -- the next `success` ping will simply have no duration recorded. This is an acceptable trade-off for avoiding database reads on the hot path.

## Graceful Shutdown

Kast enables NestJS shutdown hooks with `app.enableShutdownHooks()` at bootstrap. On SIGTERM or SIGINT:

1. All consumer groups commit their current offsets to Redpanda via `consumer.commitOffsets()` before calling `consumer.disconnect()`.
2. In-flight batch buffers in the Sink Module are flushed to PostgreSQL.
3. WebSocket connections are closed.
4. The process exits.

This ensures no events are lost or double-processed on restart. Consumer groups resume from their last committed offset. The `enableShutdownHooks` call is critical -- without it, NestJS does not intercept process signals, and consumers would disconnect without committing offsets, causing duplicate processing on the next startup.

## Consumer Tuning

All KafkaJS consumer instances share a common configuration tuned for Kast's workload:

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `sessionTimeout` | 30,000 ms (30s) | Maximum time between heartbeats before the broker considers the consumer dead and triggers a rebalance. Set higher than the default 10s to tolerate GC pauses and Sink batch flushes. |
| `heartbeatInterval` | 3,000 ms (3s) | Frequency of heartbeat messages to the broker. Standard rule of thumb: one-third of session timeout. |
| `retry` | 10 | Number of retries for transient broker errors (connection drops, leader elections) before the consumer crashes. Higher than the default to survive Redpanda rolling restarts. |

These values are configured in each consumer registration:

```typescript
{
  groupId: 'kast-sink',
  sessionTimeout: 30000,
  heartbeatInterval: 3000,
  retry: { retries: 10 },
}
```

If you run multiple API replicas, consumer groups automatically rebalance partitions across instances. The `sessionTimeout` value determines how quickly a dead replica's partitions are reassigned -- 30 seconds means at most 30 seconds of processing delay after a replica crashes.

## KafkaJS Partitioner Warning

KafkaJS v2 changed the default partitioner and logs a warning on startup if you do not explicitly choose one. Kast suppresses this warning by explicitly configuring the legacy partitioner:

```typescript
import { Partitioners } from 'kafkajs';

const kafka = new Kafka({
  // ...
  producer: {
    createPartitioner: Partitioners.LegacyPartitioner,
  },
});
```

The legacy partitioner uses the same algorithm as the Java Kafka client, ensuring consistent partition assignment when keys (monitor UUIDs) are used. This is important because all topics are keyed by monitor UUID -- changing the partitioner would break ordering guarantees for existing monitors.
